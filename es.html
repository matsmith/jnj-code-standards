    JNJ Front-end Code Standards & Best Practices

    Estándares y Prácticas Recomendadas de Desarrollo Front-end

    Información General

    Este documento contiene directrices para aplicaciones web creadas por el equipo de trabajo de Tecnología Creativa (Ingeniería de Front-end) de Johnson and Johnson US. Se creó a fin de hacerse disponible para cualquier persona que desee revisar el progreso iterativo de nuestras prácticas recomendadas. Si tiene algo que opinar, por favor, deje un comentario en el artículo del blog donde lo anunciamos.

    La motivación principal de este documento tiene dos aristas: 1) consistencia de código y 2) prácticas recomendadas. Al mantener la consistencia en los estilos y convenciones del código, podemos aliviar la carga del mantenimiento del código legado (desactualizado) y mitigar los riesgos de daño en el futuro. Adhiriéndonos a prácticas recomendadas aseguramos unos tiempos de carga de página, rendimiento y código mantenible optimizados.

    Directrices Generales

    Pilares del Desarrollo Front-end

    1. Separación de presentación, contenido y comportamiento.
    2. El etiquetado debe estar bien formado, ser semánticamente correcto y generalmente válido.
    3. Javascript debe mejorar progresivamente la experiencia.

    Prácticas Generales

    Sangrado

    Para todos los lenguajes de código, exigimos que el sangrado se realice mediante tabulaciones suaves (usando el caracter de espacio). Pulsar Tab en su editor de texto debe ser equivalente a cuatro espacios.

    Legibilidad vs Compresión

    Preferimos la legibilidad a la reducción de tamaño de los archivos cuando se trata de mantener archivos existentes. Estimulamos el uso de mucho espacio en blanco junto con arte ASCII donde sea adecuado. No es necesario para ningún desarrollador comprimir intencionalmente el HTML ni las CSS, ni ofuscar (hacer ininteligible) el JavaScript.

    Usaremos procesos del lado servidor o procesos de compilación para minimizar y comprimir en gzip automáticamente todos los archivos estáticos del lado cliente, como los archivos CSS y JavaScript.

    Etiquetado

    El primer componente de cualquier página es el lenguaje de marcado basado en etiquetas HTML. El Lenguaje de Marcado de Hiper Texto (HTML) tiene una historia sórdida pero ha logrado cumplir su cometido en los últimos años. Después de una larga experimentación con la variante XHTML basada en XML, la industria ha aceptado que el HTML es el futuro de la web.

    El etiquetado define la estructura y esquema de un documento y ofrece un contenido estructurado. El etiquetado no pretende definir la apariencia del contenido en la página más allá de rudimentarios conceptos tales como encabezados, párrafos y listas. Todos los atributos de presentación del HTML se han hecho obsoletos y los estilos deben colocarse en hojas de estilo.

    HTML5

    HTML5 es una nueva versión de HTML y XHTML. El borrador de la especificación HTML5 define un único lenguaje que puede escribirse en HTML y XML. El mismo intenta resolver los problemas encontrados en iteraciones anteriores del HTML y atiende las necesidades de las Aplicaciones Web, un área previamente no cubierta adecuadamente por el HTML. (fuente).

    Utilizaremos el Doctype HTML5 y características de HTML5 cuando sea adecuado.

    Probaremos nuestro etiquetado con el validador del W3C para garantizar que el etiquetado está bien formado. El código 100% válido no es una meta, pero la validación, sin duda, ayuda a crear sitios más fáciles de mantener así como a depurar el código. Johnson and Johnson no garantiza que el código es 100% válido, pero en cambio asegura que la experiencia a través de todos de navegadores es bastante consistente.

    Plantillas

    Para los documentos HTML5 usamos un fork de H5BP modificado según las necesidades de nuestros propios proyectos. Hazle fork al repositorio Github.

    El Doctype

    Siempre debe utilizarse un Doctype correcto que active el modo de estándares en el navegador. El modo no estándar o "Quirks" siempre debe evitarse.

    Un aspecto agradable del HTML5 es que optimiza la cantidad de código que se requiere. Los atributos sin sentido han sido eliminados y se ha simplificado considerablemente la declaración DOCTYPE. Además, no es necesario utilizar CDATA para escapar el código JavaScript incrustado, anteriormente un requisito para cumplir con el rigor del XML en el XHTML.

    Codificación de Caracteres

    Todo el etiquetado debe ser servido como UTF-8, ya que es el más amigable para la internacionalización. Debería ser asignado tanto en el encabezado HTTP como en el encabezado del documento.

    Directrices Generales de Etiquetado

    Las siguientes son directrices generales para estructurar el etiquetado HTML. Se le recuerda siempre a los autores usar etiquetado que represente la semántica del contenido en el documento que se está creando.

    • Utilice elementos P reales como delimitadores de párrafo en lugar de múltiples etiquetas BR.
    • Haga uso de DL (listas de definición) y BLOCKQUOTE cuando sea adecuado.
    • Los elementos en forma de lista siempre deben ser incluidos en UL, OL o DL, nunca en un conjunto de etiquetas DIV o P.
    • Utilice campos label para etiquetar cada campo de formulario, el atributo for debe asociarse con el campo de entrada, de manera que los usuarios pueden hacer clic en las etiquetas. Usar cursor:pointer; en el label también es sabio. nota 1 nota 2
    • No utilice el atributo size en los campos de entrada. El atributo size es relativo al tamaño de la fuente del texto dentro de la entrada. En su lugar utilice width en las css.
    • Coloque un comentario html en las etiquetas de cierre de algunos div para indicar qué elemento está cerrando. Ésto será de gran ayuda cuando haya muchos elementos anidados y sangrados.
    • Las tablas no deberían utilizarse para el diseño de la página.
    • Use microformatos y/o Microdata donde sea adecuado, específicamente hCard y adr.
    • Haga uso de las etiquetas THEAD, TBODY y TH (y del atributo Scope) cuando sea adecuado.
    • Siempre utilice texto tipo título para los encabezados y títulos. No use títulos con texto todo en mayúsculas o todo en minúsculas en el etiquetado, aplique en su lugar, la propiedad CSS text-transform:uppercase/lowercase.

    Comillas en los Atributos

    Aunque la especificación HTML5 define las comillas alrededor de los atributos como opcionales, para mayor coherencia con atributos que aceptan un espacio en blanco, todos los atributos deben estar entre comillas.

    CSS

    El segundo componente de una página web es la información de presentación contenida en las Hojas de Estilo en Cascada (CSS). La exitosa implementación de las CSS por parte de los navegadores web le ha otorgado a toda una generación de creadores web un amplio control sobre la apariencia de sus sitios web.

    Así como la información en una página web es semánticamente descrita en el etiquetado HTML, las CSS describen todos los aspectos de presentación de la página a través de una descripción de sus propiedades visuales. El poder de las CSS radica en que estas propiedades pueden mezclarse y combinarse mediante identificadores para controlar el diseño y características visuales de la página a través de la superposición de las reglas de estilo (la "cascada").

    Principios Generales de Uso

    • Añada las CSS mediante archivos externos, minimizando el número de archivos, de ser posible. Deben estar siempre en el HEAD del documento.
    • Utilice la etiqueta LINK para incluirlas, nunca use @import.
    • No incluya estilos en línea en el documento, ni mediante la etiqueta style ni directamente en los elementos. Es más difícil hacer seguimiento a estas reglas de estilo.
    • Utilice un archivo de reinicialización de CSS (como el presente en H5BP o el Reinicializador de Eric Meyer) para neutralizar la extrañas diferencias entre los navegadores.
    • Use un archivo de normalización de fuentes como YUI fonts.css
    • Lo elementos que ocurren sólo una vez dentro de un documento deben utilizar ID, de lo contrario, utilice clases.
    • Comprenda la cascada y la especificidad de selectores, de manera que pueda escribir código muy conciso y eficaz.
    • Use selectores optimizados para la velocidad. Cuando sea posible, evite selectores CSS costosos. Por ejemplo, evite el selector comodín * y no califique selectores de ID (por ejemplo, div#myid) o selectores de clase (por ejemplo, table.results.) Esto es especialmente importante en aplicaciones web donde la velocidad es de suma importancia y puede haber miles o incluso decenas de miles de elementos del DOM. Más información sobre cómo escribir CSS eficientes en el MDC.

    Modelo de Caja de las CSS

    Un profundo conocimiento y comprensión de las CSS y del modelo de caja basado en el navegador es necesario para conquistar los fundamentos de la maquetación con CSS.

    Validación de las CSS

    Normalmente, nosotros no usamos el Validador del W3C.

    Formateado de las CSS

    Como mínimo, coloque las CSS con selectores en una línea y cada propiedad en su propia línea. A las declaraciones se les aplica sangría.

    Como una mejora a ese estilo, añada 2 o 4 espacios a los estilos secundarios o relacionados. Esto permite una exploración y organización jerárquica y hace una hoja de estilos más fácil de leer (para algunas personas).

    Para entornos de múltiples autores, deben evitarse las CSS de una sola línea, ya que pueden causar problemas con el control de versiones.

    Alfabetizado

    Si está obsesionado con el rendimiento, al alfabetizar las propiedades CSS aumenta las probabilidades de que grandes patrones repetibles se hagan presentes para ayudar en la compresión GZIP.

    Classes vs. IDs

    Sólo debe asigar un atributo ID a los elementos si son únicos. Debe aplicarse a ese elemento única y exclusivamente. Las clases se pueden aplicar a varios elementos que comparten las mismas propiedades de estilo. Las cosas que deben verse y funcionan de la misma manera pueden tener el mismo nombre de clase.

    Convenciones de Nombre para los Selectores

    Siempre es preferible asignar un nombre a algo, ya sea un ID o una clase, por la naturaleza de lo que es en lugar de como se ve. Por ejemplo, un nombre de clase de textoGrandeAzul para una nota especial en una página no tiene ningún sentido si se ha cambiado a un texto pequeño de color rojo. Utilizar una convención más inteligente como textoNota es mejor, porque cuando cambia el estilo visual ésta aún tiene sentido.

    Selectores

    La especificación Selectores CSS Nivel 3 presenta un nuevo conjunto de Selectores CSS que son sumamente útiles para la mejor selección de elementos.

    Pseudoclasses

    Las Pseudoclases le permiten aplicar estilos dinámicamente al contenido. Algunas pseudoclases han existido desde las CSS1 (:visited, :hover, etc.) y las CSS2 (:first-child, :lang). Con las llegada de las CSS3, 16 nuevas pseudoclases se han agregado a la lista y son especialmente útiles para aplicar estilos a contenido dinámico. Aprenda a utilizar pseudoclases en profundidad.

    Combinadores y Selectores de Atributo

    Los Combinadores proporcionan métodos abreviados para seleccionar elementos que son elementos descendientes, elementos secundarios o hermanos de un elemento.

    Los Selectores de Atributo son excelentes para encontrar elementos que tienen un atributo específico y/o un valor específico. Conocer acerca de expresiones regulares ayuda con los selectores de atributos.

    Especificidad

    Los navegadores calculan la especificidad de un selector para determinar qué regla debe aplicarse. Si dos selectores se aplican al mismo elemento, la que tiene mayor especificidad gana.

    las IDs tienen mayor especificidad que los selectores de atributo, y los selectores de clase tienen una especificidad mayor que cualquier número de selectores de elemento. Intente siempre usar IDs para aumentar la especificidad. Hay veces cuando podemos intentar aplicar una regla CSS a un elemento y no funciona independientemente de lo que intentemos. Esto es probable debido a que la especificidad del selector utilizado es inferior a otro y las propiedades del mayor tienen precedencia sobre las que desea aplicar. Esto es más común cuando trabaja con hojas de estilos más grandes y complejas. Por lo general, no es un gran problema en proyectos más pequeños.

    Calculando la especificidad

    Cuando trabaja con una hoja de estilos grande y compleja ayuda saber cómo se calcula el valor de la especificidad del selector, para ahorrarle tiempo y hacer más eficientes sus selectores.

    La especificidad se calcula contando los diversos componentes de sus estilos CSS y expresándolo en una forma (a, b, c, d).

    • Elemento, Pseudoelemento: d = 1 – (0,0,0,1)
    • Clase, Pseudoclase, Atributo: c = 1 – (0,0,1,0)
    • Id: b = 1 – (0,1,0,0)
    • Estilo en línea: a = 1 – (1,0,0,0)

    Sin embargo, sería mejor utilizar una calculadora de especificidad.

    Pixels vs. Ems

    Utilizamos la unidad de medida px para definir font size, ya que ofrece un control absoluto sobre el texto. Nos hemos dado cuenta de que utilizar la unidad em para el tamaño de fuente solía ser popular para adaptarse a que el Internet Explorer 6 no puede cambiar el tamaño de texto basado en píxeles. Sin embargo, todos los principales navegadores (incluyendo IE7 e IE8) ahora soportan el cambio de tamaño del texto en unidades de píxeles y/o el zoom de página completa. Dado que IE6 en gran medida es considerado obsoleto, es preferible el tamaño en píxeles. Además, el line-height sin unidad establecida es preferible porque no hereda un valor de porcentaje de su elemento principal, sino que en cambio, se basa en un multiplicador del font-size.

    Notación abreviada

    En general, se prefiere la notación abreviada de las CSS debido a su brevedad y la capacidad para luego volver atrás y agregar valores que ya están presentes, como es el caso de margin y padding. Los desarrolladores deben estar conscientes de las siglas ADAI (o TRBL por sus siglas en inglés), que indican el orden en el que se definen los lados de un elemento en el sentido de las agujas del reloj: Arriba, Derecha, Abajo, Izquierda. Si bottom no está definido, hereda su valor de top. Asimismo, si left no está definido, hereda su valor de right. Si sólo se define el valor top, todos los lados lo heredan de esa declaración.

    Estilos Generales de Texto y Fuentes

    Encabezados

    • Defina el estilo predeterminado para los encabezados h1-h6 incluyendo los encabezados que son enlaces. Es útil declararlos en la parte superior del documento CSS y modificarlos cuando sea necesario para mayor coherencia a través del sitio.
    • Los encabezados deben mostrar una jerarquía que indica los diferentes niveles de importancia de arriba a abajo empezando por h1 con el mayor tamaño de fuente.
    • SEO: Para obtener una idea aproximada de cómo es organizada y leída la jerarquía de su página, utilice la barra de herramientas para desarrolladores a fin de deshabilitar los estilos CSS. Terminará con una vista basada en texto de todas las etiquetas h1-h6, strong, em, etc.

    Enlaces

    • Los estilos predeterminados para los enlaces deben ser declarados y deben ser diferentes a los estilos del texto principal y con estilos diferentes para el estado hover.
    • Cuando aplique estilos a los enlaces con subrayado, utilice border-bottom y algo de padding con text-decoration: none;. Esto simplemente se ve mejor.

    Tipografía Web

    El uso de fuentes y tipos de letra personalizados en la web ha ido creciendo en popularidad en los últimos años. Con un soporte nativo por parte de los navegadores en ascenso y varios servicios y APIs disponibles ahora, existe un impulso real en este espacio. Cada uno de estos enfoques tienen sus propias ventajas y desventajas. Antes de que un proyecto arranque es mejor realizar algo de investigación sobre las limitaciones tecnológicas y de licencias a fin de elegir el enfoque adecuado para el proyecto específico.

    Todos estos enfoques tienen inconvenientes en cuanto a código adicional, tiempo de desarrollo y rendimiento (reloj y percibido). Familiarizarse con estos temas y comunicarlo a los demás miembros del equipo y al cliente ahorrará importantes problemas más adelante en el proyecto.

    Aquí encuentra listados algunos métodos populares para incrustar fuentes personalizadas, organizados en orden de nuestra preferencia en su aplicación.

    @font-face

    La regla @font-face le permite definir fuentes personalizadas. Ésta fue definida inicialmente en la especificación CSS2, pero fue eliminada de la CSS2.1. Actualmente, es un proyecto de recomendación para CSS3.

    Nuestra primera y preferida elección para personalizar las fuentes en la web es @font-face, simplemente porque es parte del proyecto de trabajo del Módulo de Fuentes de CSS, lo que significa que seguirá creciendo en popularidad a medida que crece el soporte de los navegadores y su facilidad de uso mejora conforme se vuelve más estable.

    Por ahora, cuando utiliza @font-face es recomendado declarar el origen para cada formato de la fuente. Esto es importante si desea trabajar con el mayor número de navegadores, aunque no es un requisito para su uso.

    Los formatos de fuente incluidos en la especificación son:

    • woff: WOFF (Web Open Font Format)
    • ttf: TrueType
    • ttf, otf: OpenType
    • eot: Embedded OpenType
    • svg, svgz: SVG Font

    Typekit

    Utilizar Typekit tiene sus ventajas y no debería ser ignorado por completo a la hora de elegir el método a utilizar para añadir fuentes personalizadas a un sitio web. Cuenta con una plataforma de integración fuerte y es un servicio escalable y popular. Se puede utilizar con Google WebFonts y se puede agregar fácilmente a WordPress, Posterous, Typepad, y otros sitios manejados por CMSes similares.

    Sin embargo, la plena utilización de TypeKit no viene sin un costo. Si usted necesita usarlo en más de 2 sitios o en un sitio que recibe una gran cantidad de pageviews tendrá que pagar un costo anual de $49.99, y para los sitios con más de un millón de pageviews le cuesta el doble. No obstante, es probable que tenga el dinero para cubrir el costo si usted está recibiendo más de un millón de pageviews. Si no, entonces puede que usted deba reconsiderar su modelo de negocios.

    Pros
    • Gran biblioteca de fuentes, incluyendo fuentes de Adobe
    • Fácil implementación
    • Integración con la Webfont API de Google y con plataformas de blogs
    • El plan gratuito tiene límites, pero no caduca
    Contras
    • Requiere Javascript para utilizarlo
    • Accesso a biblioteca de fuentes limitada sin pagar
    • Los planes libres y más baratos sólo permiten el uso en 1-2 sitios web y 2-5 fuentes por sitio
    • Tiene que pagar para usarlo en más de 1 sitio

    Scalable Inman Flash Replacement (sIFR)

    No le recomendamos que utilice este método, pero debido a que es ampliamente utilizado sentimos que era necesario incluirlo, por lo que puede tomar una decisión bien informada al elegir el método que debe usar con las fuentes web personalizadas.

    A pesar de su gran popularidad entre los diseñadores web, así como su decente soporte en la mayoría de los navegadores, los inconvenientes en su uso superan a su conveniencia. La razón más grande y más obvia para no utilizar sIFR es el hecho de que utiliza Flash. Además, para que el flash funcione, se requiere JavaScript y las secuencias de comandos deben cargarse antes que el texto en el que lo use sea visible en la página. Sin mencionar que aumenta el tiempo de carga de la página y puede causar que un sitio lento se haga aún más lento.

    JavaScript

    El código JavaScript, cuando se aplica correctamente a una página web, mejora la experiencia de usuario global y basada en navegador a través de la asociación a eventos y el control de la capa de comportamiento en general.

    JavaScript ha visto una explosión en popularidad en los últimos años debido a las potentes nuevas implementaciones de los navegadores que han facultado finalmente la creación de aplicaciones totalmente basadas en el navegador web. Además, un uso cuidadoso de JavaScript permite la completa manipulación y control sobre los otros dos componentes de la creación de páginas web, el etiquetado HTML y las CSS. Hoy en día la estructura de las páginas y los estilos visuales de las páginas pueden ser manipulados en tiempo real sin necesidad de actualizar la página web completa.

    Bibliotecas JavaScript

    Principalmente desarrollamos nuevas aplicaciones en jQuery, aunque tenemos un gran dominio del JavaScript normal, así como de todas las bibliotecas modernas de javascript más importantes.

    Principios Generales de Creación de Código

    • 99% del código debe estar alojado en archivos JavaScript externos. Deben ser incluidos al FINAL de la etiqueta BODY para obtener un rendimiento máximo de página.
    • No se base en la cadena de agente de usuario (user-agent). Haga una adecuada detección de características. (Más información en Dive Into HTML5: Detection y la documentación jQuery.support)
    • No utilice document.write ().
    • Todas las variables booleanas deben comenzar con "is".
    • Nombre las variables y funciones lógicamente. Por ejemplo: popUpWindowForAd en lugar de myWindow.
    • No minimice manualmente. A excepción de la tradicional i, etc. para bucles for, las variables deben ser lo suficientemente largas para que tengan sentido.
    • La documentación debe seguir la estructura de NaturalDocs.
    • Las constantes o variables de configuración (como la duración de las animaciones, etc.) deben estar en la parte superior del archivo.
    • Esfuércese por crear funciones que puedan ser generalizadas, tomar parámetros y devolver valores. Esto permite la reutilización sustancial del código y cuando se combina con includes o scripts externos, puede reducir la sobrecarga cuando los scripts tienen que cambiar. Por ejemplo, en lugar de escribir directamente el código de una ventana emergente con el tamaño de la ventana, opciones y url, considere el crear una función que tome el tamaño, url y opciones como variables.
    • ¡Comente su código! Ayuda a reducir el tiempo dedicado a solucionar problemas de funciones de JavaScript.
    • No pierda su tiempo con comentarios <!-- --> en torno a su javascript incrustado, a menos que se preocupe por Netscape 4. :)
    • Organice su código como un Objeto Literal/Singleton, en el Patrón de Módulo, o como un Objeto con constructores.
    • Minimice las variables globales. Mientras menos globales cree, mejor. Generalmente una para el namespace de su aplicación, es un buen número.

    Rendimiento

    A medida que continuamos forzando los límites de lo que puede hacer la web, sigue siendo igual de importante que una página web se pueda utilizar con un mínimo esfuerzo o tiempo de espera. La siguiente sección explica cómo pueden ser optimizadas las páginas web para mantener contentas a todas las audiencias.

    Optimizar la entrega de CSS y JavaScript

    Hay muchas optimizaciones que deben hacerse para servir CSS y Javascript en Producción:

    • Siga las Directrices de Rendimiento de Yahoo.
    • Optimice las imágenes utilizando smush.it. También puede usar YSlow para que auto-optimice todas sus imágenes por usted.
    • Establezca los encabezados de caché adecuadamente.
    • Considere la posibilidad de un subdominio sin-cookies para recursos estáticos.
    • Las CSS deben estar ubicadas en el <head> del documento, Javascript debe estar justo antes de la etiqueta </body>.
    • Tanto las CSS como el JavaScript deben servirse minimizados. La mayoría de las personas utiliza el compresor de YUI para ello.
    • Ambos deben ser servidos usando gzip en el cable.
    • Las CSS deben ser concatenadas. Obviamente esto sólo es posible para los archivos que comparten el mismo tipo de media (por ejemplo, screen o print). El objetivo general aquí es reducir el número de conexiones HTTP a las dependencias durante la carga de la página.
    • Los guiones JavaScript deben ser concatenados. Aunque un gestor de dependencias de guiones de ajax sería ideal (similar al YUI Loader 3), éste es bastante complicado de implementar. En su lugar, yo recomendaría una sola descarga para la mayoría de los scripts utilizados en el sitio. (Por supuesto, el almacenamiento adecuado en caché debe utilizarse para conservar el archivo tanto tiempo como que sea razonable).
    • La concatenación y minimización normalmente se producen durante un proceso de generación automática, al empaquetar el código para su distribución al servidor de ensayo o de producción. Muchos usan herramientas como Ant o Maven para ello.
    • Evite bloques de <script> incrustados directamente en el código HTML. Éstos bloquean el despliegue del contenido en el navegador y son devastadores para el tiempo de carga de la página.

    Optimizar la ejecución de JavaScript

    Durante la carga de una página normalmente hay un montón de guiones esperando ejecutarse, pero usted puede priorizarlos. Este orden se prioriza en función de la respuesta del usuario:

    1. Los guiones que cambian la naturaleza visible de la página necesitan ejecutarse absolutamente de primeros. Esto incluye cualquier ajuste de tipo de fuente, disposición visual de las cajas, o pngfixes de IE6.
    2. Inicialización de contenido de página.
    3. Inicialización de encabezado, navegación superior y pie de página.
    4. Adición de controladores de eventos.
    5. Omniture, Doubleclick y otros guiones de terceros.

    Formatos de imagen

    Hay cuatro formatos de imagen principales que se deben utilizar, detallados a continuación:

    JPEG

    Este cubrirá todas las imágenes basadas en fotografías, como imágenes promocionales de la página principal y de categorías o cualquier cosa con un gran número de colores.

    PNG24

    Este formato, de fácil acceso en Photoshop, genera imágenes de gran número de colores y soporta transparencia gradual para cada píxel. Relativamente, es un formato bastante pesado en cuanto a su peso en kilobytes. Es el único formato que IE6 necesita para aplicarle un pngfix. En este caso, Johnson and Johnson recomienda el guión DD_belatedPNG (Un pngfix corrige el problema por el que los PNG24 parecen tener un fondo gris o azul claro en IE6. Éstos no son siempre compatibles con la propiedad background-position).

    En muchos casos, puede utilizar un GIF de reserva para IE6, en lugar de un PNG24. Esto es especialmente cierto si cualquier sprite debe realizarse en PNG24. Todos los pngfixes son muy lentos y costosos, por lo que es mejor evitar usarlos.

    PNG8

    Una sorprendente diversidad de colores pueden ser capturados dentro de 256 colores, por lo que vale la pena tratar con PNG antes de enfocarnos en JPG. PNG también se puede comprimir mucho más que GIF (usando herramientas como pngcrush y pngquant). Este formato permite transparencia gradual en casi todos los navegadores, pero en IE6, los píxeles semi-opacos sólo aparecen 100% transparentes. En muchos casos esto es suficiente. Adicionalmente no requiere la ejecución de un guión pngfix, por lo que está optimizado para la velocidad.

    Photoshop no puede salvar estos archivos semi-opacos correctamente, pero Fireworks puede hacerlo. Más detalles en http://www.sitepoint.com/blogs/2007/09/18/png8-the-clear-winner/

    GIF Transparente 89a

    GIF 89ª ofrece la flexibilidad de la transparencia y un amplio soporte en los navegadores, pero las limitaciones de no tener ni transparencia gradual, ni una profundidad de color por encima de 256. Según nuestra experiencia, profundidades de color de 64 todavía proporcionan miniaturas de muy buena calidad y mantienen el tamaño del archivo comparativamente muy pequeño.

    Todas las imágenes de pocos colores como iconos o gráficos temáticos deben crearse en PNG8, ya que es el más eficiente de estos cuatro formatos con respecto al tamaño . PNG8 es nuestra recomendación principal para la mayoría de las imágenes de un sitio.

    Puede encontrar información detallada sobre el formato PNG, soporte en navegadores y los pros y contras de cada formato en este excelente artículo.

    Para una mayor optimización de todos estos formatos, pase las imágenes a través de Smush.it de Yahoo y le mostrará como pueden reducirse aún más.

    Almacenamiento en Caché

    Para el contenido estático, el navegador debe almacenar el archivo localmente en caché, siempre y cuando sea razonable. El contenido que debe tener el almacenamiento en caché establecido a futuro incluye cosas como:

    • Archivos CSS y JavaScript
    • Imágenes de los productos
    • Gráficos temáticos
    • favicon.ico
    • Películas .swf de flash y videos
    • Imágenes promocionales (probablemente un almacenamiento más ligero)

    Para mejor almacenamiento en caché, aproveche el Expires http header. Éste se trata de un encabezado de expiración en el futuro lejano, que le dice al navegador que esta respuesta no será obsoleta hasta el 15 de abril de 2015.

    Expires: Thu, 15 Apr 2015 20:00:00 GMT

    Si su servidor es Apache, utilice la directiva ExpiresDefault para establecer una fecha de caducidad en relación con la fecha actual. Este ejemplo de la directiva ExpiresDefault establece la fecha de expiración a 1 año desde el momento de la solicitud.

    ExpiresDefault "access plus 1 year"

    Expires http header se debe establecer en un valor comprendido entre un mes desde el presente hasta un año (futuro lejano) del presente. El almacenamiento en caché sólo se aplica a la dirección URL exacta, por lo que un cambio en el nombre de archivo de un recurso creará una nueva copia. Muchos desarrolladores utilizan un proceso de compilación para añadir un número de versión o marca de tiempo (timestamp) a sus recursos. Cada compilación posterior iniciará una nueva versión en caché, permitiéndole utilizar fechas de caché de futuro lejano sin preocuparse. Google tiene muchos más detalles acerca de la memoria caché del navegador.

    Fragmentación de recursos entre dominios

    Sin dudas, el contenido estático debe ser servido desde un dominio diferente al que sirve el HTML. Esto es lo óptimo para que no haya ningún encabezado de cookies extra en todas las solicitudes de contenido estático. También es mucho más fácil escribir reglas de caché para todo el dominio. (cualquier subdominio del dominio actual va a heredar las cookies del dominio, por lo que vale la pena usar un dominio completamente nuevo).

    Sin embargo, con suficientes recursos (especialmente imágenes) el número de solicitudes crece lo suficiente como para disminuir la carga de la página. Muchos navegadores tienen un límite muy bajo en cuanto al número de recursos que se descargan simultáneamente por dominio. (En IE6 y 7, es sólo dos). En este caso, podemos servir los recursos desde varios subdominios como:

    • estatico1.otrodominio.com
    • estatico2.otrodominio.com
    • estatico3.otrodominio.com

    Más información sobre fragmentación entre dominios en Google Speed

    Evite IFRAMEs

    Los iframes son el elemento más costosos que puede añadir a una página determinada. Bloquean el arranque del evento onload hasta que se hayan completado todas las páginas. A veces son útiles para que otra agencia gestione los scripts de seguimiento. Por ejemplo, la etiqueta floodlight de DoubleClick es un iframe y el administrador puede agregarle píxeles de seguimiento desde su panel de control. En cualquier caso en que se añade un iframe, se debe anexar al DOM de forma dinámica después de que window onload se ha desencadenado. Más detalles en el Sitio de Rendimiento de Yahoo.

    Integración de terceros

    Omniture

    Recomendamos añadir el código JavaScript de Omniture al DOM utilizando JavaScript después de que la página se ha cargado (ya sea un evento DOM ready o window load). Usando esta técnica, no hay ninguna dependencia de un script en un dominio externo que puede retrasar (y potencialmente congelar) la carga de la página.

    Flash

    El contenido HTML de reserva debe estar en el lugar de Flash en todos los casos para maximizar el valor SEO. Para el contenido Flash basado en XML, el contenido HTML de reserva debe aprovechar exactamente el mismo archivo XML, para garantizar la consistencia de los datos.

    Todas las sustituciones de contenido se deben realizar con SWFObject, pero sin las etiquetas de script incrustadas directamente en el código. La inicialización del SWFObject debe iniciarse después del evento DOM Ready. La versión mínima del reproductor de Flash debe establecerse como valor mínimo a la v9, para asegurar la compatibilidad con AS3.

    Estrategia de Rendimiento Multi-Navegadores

    Hay dos verdades importantes cuando se trata de la experiencia en el navegador:

    1. Todo el mundo quiere la experiencia con la mejor capacidad de respuesta posible.
    2. Todo lo agregado a la página la hace más lenta.

    Así que con estas dos realidades, ¿qué medidas debemos tomar para que todos sean felices?

    Crear indicadores de éxito con el cliente

    Éstos deben ser adaptados al cliente, al proyecto y creados antes de la fase de wireframing. Estos objetivos deben ser razonables desde un punto de vista técnico, así como comprobables.

    Algunos de los objetivos que serían apropiados:
    1. El navegador más lento soportado debe ir de una memoria caché vacía a una totalmente cargada e inicializarse en un plazo de 5 segundos.
    2. Los estados Hover (y otros cambios "instantáneos") deben responder dentro de 100ms.
    3. Las animaciones deben verse fluidas, pudiendo producir saltos < 15% de las veces (entre todos los navegadores).

    Para los objetivos basados en el tiempo de carga, es importante definir el sistema de referencia. Algo así como PageTest es una buena opción. Además, los objetivos pueden ser definidos para varias páginas, por ejemplo: las dos páginas de destino más populares (por ejemplo, la página principal y soporte).

    Si el cliente tiene metas más agresivas que sean razonables con el diseño previsto, las expectativas deben ser establecidas en todos los ámbitos, informando con antelación al equipo que las metas de rendimiento van a ser primordiales.

    Comunicar el impacto en el rendimiento durante la fase de diseño

    Internamente

    Durante la AI, el IxD y el diseño visual, es rol del ingeniero de front-end comunicar el impacto en el rendimiento de las características interactivas o determinadas técnicas visuales en los navegadores elegidos. Establézcale límites a los diseñadores: "Si estamos utilizando Cufon, no podemos tener más de 10 elementos con fuentes personalizadas por página".

    Externamente

    Es necesario establecer la expectativa de que No todos los navegadores tendrán la misma experiencia. Ninguno tendrá el mismo desempeño que los otros, ni puede tener sentido el tener características idénticas. Sería sensato eliminar algunas características de la experiencia de Internet Explorer 7. Algunas características que podrían considerarse para su eliminación son: sombras, transiciones, esquinas redondeadas, transparencia y degradados.

    Al comunicar el impacto de algo:
    • Aclare el impacto con tanto detalle como sea posible: "afectará la carga de la página" vs "añadirá 2 segundos a la carga de la página en IE"
    • Proporcione una PDC (prueba de concepto) rápida si es razonable: "Esta página de aspecto similar, sin siFR carga en 2 segundos, con siFR carga en 8 y tarda en mostrarse mientras desplazamos la página"

    Desarrollar según las prácticas recomendadas

    Elija bibliotecas y plugins que estén optimizados para el rendimiento. Tome decisiones de arquitectura acertadas basándose en los objetivos de rendimiento. Reduzca al mínimo la manipulación del DOM cuando sea posible y escriba estilos para evitar cambios visuales a la página mientras carga y se inicializa.

    Medir el desempeño durante el Control de Calidad

    Los equipos de Control de Calidad deben priorizar también los tickets relacionados con el rendimiento junto con los de problemas visuales, funcionales y de usabilidad. Los desarrolladores y Control de Calidad deben determinar cómo se asignará esa prioridad. Durante el Control de Calidad, los indicadores de éxito deben probarse periódicamente.

    Herramientas para elaborar pruebas
    • YSlow, Page Speed, Hammerhead, MSFast, PageTest
    Cuando no se cumplen los objetivos de rendimiento, hay tres opciones:
    1. Reconstruya el código - elabore un perfil, descubra cuellos de botella, re-escriba el código, optimice para lograr una ejecución más rápida en el navegador
    2. Elimine la característica - desactivela sólo para navegadores más lentos
    3. Rediseñe el enfoque de la IU - quizás el diseño podría utilizar un ajuste que podría eludir el problema completamente

    Con este enfoque, creemos que todas las partes tienen una mejor oportunidad de tener las expectativas alineadas funcionando, así como un flujo de trabajo más razonable para enfrentar los retos del rendimiento.

    Pruebas y Soporte de Navegadores

    La audiencia de hoy puede elegir entre toda una gran cantidad de navegadores web, cada uno con una experiencia ligeramente (o dramáticamente) diferente. Como desarrolladores, es nuestra responsabilidad elegir cómo se muestran a los usuarios las páginas web que construimos. En esta sección se describe cómo nosotros, en Johnson and Johnson, tomamos algunas de estas decisiones clave.

    Que soportamos

    Johnson and Johnson soporta cualquier navegador con grado A en el Soporte Graduado de Navegadores de Yahoo, con la excepción de Opera. Puede haber excepciones, teniendo en cuenta los mercados regionales y sus indicadores particulares.

    Haremos todo lo posible para soportar cualquier otro navegador de misión crítica exigido por el cliente (Internet Explorer 5.5, Opera, Konqueror, Safari 3 para PC, etc), aunque no podemos garantizar que toda la funcionalidad esté disponible.

    Como realizamos las pruebas

    Las realización de pruebas integrales de navegador son imprescindibles en cada proyecto web de Johnson and Johnson. Debe hacerse un esfuerzo considerable para hacer las pruebas entre múltiples navegadores y plataformas a fin de garantizar la calidad y una experiencia de usuario consistente. La creación de un entorno de pruebas puede ser un desafío, pero bien vale la pena hacerlo.

    En Microsoft Windows

    Pruebas de IE

    Dado que es imposible tener más de una sola copia de Microsoft Internet Explorer instalada en un PC, las pruebas de IE son un reto. Afortunadamente, Microsoft finalmente ha publicado versiones de desarrollo de Internet Explorer antiguos disponibles para su descarga. Estos discos duros virtuales (VHD por sus siglas en inglés) ejecutan versiones simplificadas de Microsoft Windows que expiran en el tiempo. Normalmente hay que reconfigurarlas luego de algunos meses. Las copias de desarrollo de Microsoft Windows de su licencia de MSDN (si están disponibles) también pueden ser una opción, dependiendo de a que tiene acceso.

    • Virtual PC - Virtual PC debe estar instalado en el equipo, y si está en Windows 7, debe utilizar "Modo XP".

    Revisiones de Código

    Una revisión de código es la piedra angular del proceso formal para garantizar la calidad de las experiencias de usuario desarrolladas por el equipo de Tecnología Creativa. Se trata de una reunión entre los autores del etiquetado, los revisores y otras partes interesadas, con el aporte esperado y posteriores revisiones de código. En pocas palabras, promovemos la conducción de revisiones de código para mantener nuestras herramientas nítidas y limpias.

    ¿Por qué debería hacer una revisión de código?

    Las revisiones de código son una inversión estratégica de tiempo para mitigar el riesgo en un proyecto.

    Muchas veces, a los desarrolladores de la interfaz se les pide que creen el etiquetado a partir de los wireframes o composiciones visuales. Sin embargo, es posible que las pantallas que se han diseñado no se puedan traducir al etiquetado con facilidad o sin comprometer la calidad. Las revisiones de código brindan una oportunidad para que este riesgo sea abordado y resuelto antes de que comience la producción completa de las páginas.

    Las revisiones de código aumentan el nivel general de conocimiento entre los proyectos

    Dado que las revisiones de código involucran a miembros de dentro y fuera de un equipo de proyecto, las técnicas y prácticas recomendadas son compartidas fácilmente a lo largo de todo el equipo de Tencnología Creativa.

    Las revisiones de código eliminan los errores antes de que se reproduzcan de unas pocas plantillas a múltiples páginas

    Idealmente, las revisiones de código se llevan a cabo al inicio del proceso de desarrollo, antes de que comience la producción completa de las páginas. Cuando las plantillas son revisadas ​​por el equipo y pasadas a través de las herramientas de validación y navegadores, los errores pueden y van a aparecer. Este es el momento ideal para que los errores se corrijan.

    Las revisiones de código le dan a un conjunto de ojos frescos la oportunidad de detectar problemas en el código

    Los revisores de fuera de un equipo de proyecto pueden detectar problemas con el código más fácilmente que los autores del etiquetado, quienes han estado trabajando con el código por un largo período de tiempo.

    ¿Quién debe participar en una revisión de código?

    En última instancia, se requiere que el lider de Ingeniería de Front-end en un proyecto garantice que se sigan los procedimientos apropiados de revisión de código.

    Idealmente, el lider del equipo debe actuar como facilitador para una sesión de revisión de código, a menos que el lider del equipo sea tambien el desarrollador de la interfaz cuyo código se encuentra bajo revisión. En esta situación, un gerente de proyecto puede integrarse como facilitador.

    El equipo de revisión debe incluir al menos dos miembros senior del equipo de Tecnología de Interfaz versados ​​en desarrollo y prácticas recomendadas.

    ¿Qué se requiere para una revisión de código?

    Antes de que se lleve a cabo una revisión de código, las plantillas a ser revisadas deben estar completamente desarrolladas, validadas y probadas en los navegadores y plataformas requeridas por el proyecto.

    Un lider de equipo y/o desarrollador de interfaz debe distribuir las siguientes 48 horas antes de la fecha de la revisión de código:

    • El código para todas las páginas, server-side includes asociados, CSS y JavaScript. Éstos deben estar completamente comentados, impresos con los números de línea a lo largo del lado izquierdo y el nombre del archivo/página en el pie de cada página impresa.
    • Capturas de pantalla de cada plantilla
    • Direcciones URL de las plantillas, si procede
    • Una lista de navegadores y plataformas soportadas por el proyecto
    • Una lista de problemas conocidos/áreas de interés

    Es típico que el código deba cambiar constantemente hasta que la revisión del código se lleve a cabo. Desafortunadamente, esto no deja tiempo suficiente para la validación y las pruebas. Si este es el caso, se recomienda que la revisión del código se aplace para una fecha que asegure una revisión del código adecuada.

    Además, un lider de equipo y/o desarrollador de interfaz debe reservar una sala de conferencias y un número de llamada para todos los asistentes, ya que es posible que miembros del equipo del proyecto o del equipo de revisión de código estén fuera del sitio. Una hora debería ser suficiente tiempo para la revisión de dos o tres plantillas, sin embargo, el tiempo variará dependiendo del volumen y la complejidad de las plantillas.

    Durante la revisión de código, un lider de equipo y/o desarrollador de interfaz debe conducir la reunión, mientras que el lider de equipo o gerente del proyecto toma notas y asigna puntos de acción. Los revisores deben haber revisado el código con anterioridad y asistir preparados para hacer preguntas o proporcionar feedback.

    Las notas y puntos de acción deben ser distribuidos a todos los asistentes después de la revisión del código. Si resultan cambios sustanciales de una revisión de código, o no se revisa todo el código, puede ser necesario programar una segunda revisión del código. Sin embargo, esto debe ser discutido entre el equipo del proyecto para determinar su viabilidad.
